# v3: multi-sample determinization voting — implement, verify, ship decision — 2026-07-23

Implements the top-ranked candidate from `docs/tie_break_v2_2026-07-23.md`'s own "suggested
next step": instead of scoring every root option under a single arbitrary hidden-world guess
(`sample_determinization()` called once per decision, unchanged since v1), score every option
under **N=8** independent hidden-world samples and pick by majority vote (score-sum as
tie-break across samples, falling through to v2's target-aware `_break_ties` for any residual
exact tie). v2's tie-break code was already committed but never shipped; it ships aboard this
release as the fallback layer.

## A0. Time-budget math (hard prerequisite)

Computed by `tools/compute_v3_sample_budget.py`, from real data on disk, not guesses:

- Real Kaggle episode config (75 real episodes, `runs/our_episodes/`): `actTimeout: 0`
  (per-action timeout **disabled**), `runTimeout: 2000` (whole-*episode* budget, not per-move).
- Real single-determinization `choose_action()` cost (`runs/near_tie_measurement/replays.jsonl`,
  2340 samples): mean 60.3ms, median 43.8ms, **p90 121.4ms, max 357.8ms**.
- Real max decisions/game (800 real games, `runs/kernel_vs_baseline/*/vs_baseline_trace.jsonl`):
  **max 91**, p90 49, median 31.
- No search-tree reuse across determinizations is possible (`search_begin` bakes the hidden
  world into the session at construction) — N-sample cost ≈ N × single-sample cost.
- Worst case: `N × 357.8ms × 91 ≤ 50% of 2000s = 1000s` → **N ≤ 30**. Comfortably above the
  N<5 stop condition specified in the task.
- **Chosen N=8** (3.8× margin below the ceiling), with a **5s self-imposed anytime time-guard**
  per decision (Kaggle imposes none) as a safety net against pathological board states.

## A1. Implementation (`agents/search_scorer.py`)

- New `_score_world()` (extracted from the old single-world body of `choose_action`): scores
  every option under one sampled world, **including within-world tie-breaking via `_break_ties`
  before that world casts its vote**. This was a real bug caught and fixed during
  implementation, not present in the final code: an early version picked each world's plain
  `max()`-scoring option without applying `_break_ties` first. Because many real ties are
  *structurally* symmetric (e.g. attaching energy to interchangeable bench Pokemon) and don't
  depend on hidden information, they reproduce identically across every sampled world — without
  per-world tie-breaking, such a tie would resolve to the same arbitrary first-list-order option
  in all 8 worlds, produce a clean 8/8 majority vote, and **never reach `_aggregate_votes`'s own
  tie-break fallthrough at all**, silently reintroducing the exact bug v2 fixed. Caught via an
  end-to-end check against the 3 real v2 repro cases (see Verification below) before any gate
  spending.
- New `_aggregate_votes(votes, score_sum, selection)`: majority vote → score-sum tie-break →
  `_tie_break_key` fallthrough (v2, unmodified). Pure function, no engine access — directly
  unit-tested with synthetic vote data.
- Time-guard: checked only *between* fully-completed worlds (never mid-option-loop), so no
  world ever contributes a partial vote; always ≥1 completed world.
- `evaluate()` untouched, exactly as with v2.
- `trace_fn` extended with `n_samples_completed`, `votes`, `score_sum`, `time_guard_fired`.

**Tests** (`tests/test_v3_voting.py`, 5 cases): clean majority win, vote-tie→score-sum
tie-break, double-tie→v2 kind-priority fallthrough, double-tie→v2 pipeline-proximity
fallthrough, and a real-decision time-guard test (injected per-world delay, confirms the guard
fires and still returns a valid decision from fewer than N worlds). All pass, plus the full
existing suite (`test_tie_break.py`, `test_obs.py`, `test_baseline.py`, `test_encode.py`,
`test_search_scorer.py`).

**End-to-end confirmation**: re-ran the live `choose_action()` (not just the isolated
`_break_ties` call `test_tie_break.py` uses) against all 3 of v2's real repro cases — all 3
still resolve to the correct pipeline-aware target, confirming the within-world tie-break fix
actually protects v2's behavior inside the new voting path.

## A2. Causal smoke check — and a real methodology correction

Re-ran `tools/measure_near_tie_hypothesis.py`'s replay-and-vote machinery (110-117 real decline/
starvation decisions, 20 replays each) at N=1 and N=8 via new `tools/run_flip_rate_smoke_check.py`.

**The task's literal target ("flip rate <20% vs. current 93.6%") could not be met, by any
possible agent** — and investigating why surfaced a real flaw in that metric, not in v3. The
existing "flip rate" (`tools/analyze_near_tie_results.py`) measures how often a replayed
decision's *chosen kind* differs from what the **historical ladder game actually did** — not
self-consistency across repeated replays. Many of the decline-set decisions get a decisive,
*consistent* preference for attack now (one real example: attack scored 125.6 vs. the
historically-chosen attach's 91.6, chosen 8/8 internal votes, 10/10 repeated top-level replays)
— which is the agent correctly and repeatably disagreeing with an old, likely-mistaken
historical choice, not noise. That will register as a "flip" under this metric regardless of
how stable the agent becomes, so it structurally cannot collapse toward 0 for *any* agent.

The metric that actually measures A2's intent — self-agreement across repeated replays of the
same board state, independent of history — tells a different, real story:

| | mean self-consistency | fully unstable (mode agrees <50%) | perfectly consistent (20/20) |
|---|---|---|---|
| N=1 | 95.8% | 1.7% | 84.6% |
| **N=8 (v3)** | **97.3%** | **0.0%** | **90.6%** |

A real, directionally-correct improvement — modest in size because the true baseline
instability was already fairly low, not because the fix doesn't work. (User-confirmed decision
to proceed to A3 on this corrected basis, given the literal metric can't move as specified.)

## A methodology finding that applies to the whole project, not just this cycle

While pairing the v1-vs-baseline and v1-vs-v3 comparisons by seed, direct verification showed
**the "seed" argument threaded through `tools/eval_arena.py --seed` / kernel drivers'
`SEED_BASE` does not control game-engine determinism at all**: `cg.game.battle_start()` takes
no seed parameter, and the engine's own hidden-state generation (deck shuffle, coin flip, draws)
is independent of Python's `random.seed()`.

Confirmed two ways: same v1 code, same nominal seeds 0-399, local vs. kernel infra —
**51.2% game-outcome agreement** (chance level for a ~43% win-rate process). Same v1 code, same
nominal seeds 0-19, **local vs. local** (two separate runs, same machine) — **30% agreement**.
Not a cross-infra artifact; fundamental to the engine.

This means every "paired by seed" comparison in this project's history (including the v1-vs-
baseline / v2-vs-baseline gates already on record) was never actually paired — each 400-game run
is a set of genuinely independent games. **This does not retract any prior conclusion**: the
statistical method already used throughout (report each run's own win rate + its own 95% CI,
compare via CI overlap) is the *correct* treatment for independent samples regardless, and
that's what was actually computed and reported every time — only the "paired/seed-matched"
narrative was incorrect, not the win-rate numbers or the CI-overlap-based conclusions drawn from
them. Recorded as a new method lesson for Workstream C's CLAUDE.md compaction.

## A3. 400-game gate

**Local legs** (`tools/eval_arena.py`, ≤2 workers, `tools/loss_review.py` for mechanism —
unmodified in both cases):

| | v1 (local) | v3 (local) |
|---|---|---|
| win rate | 180W/220L = **45.00%** | 167W/233L = **41.75%** |
| 95% CI | [40.1%, 49.9%] | [36.9%, 46.6%] |
| attack-decline rate | 74/(losses) = 9.2% | 61/(losses) = **7.2%** (better) |
| starvation rate | 179/1096 = 16.3% | 215/1122 = **19.2%** (worse) |
| tied-and-lost, of starvation | 73/179 = 40.8% | 80/215 = **37.2%** (better, as a rate) |
| tied-and-lost, absolute count | 73 | 80 (worse, in absolute terms) |
| evolve-decline rate | 51/228 = 22.4% | 45/236 = **19.1%** (better) |
| near-tie rate (top-2 within 5%) | 4562/5144 = 88.7% | 4786/5485 = 87.3% (roughly flat) |
| time-guard activations | n/a | **0/9843** (0.00%) — the 5s cap never fired |

CIs overlap substantially — win rate is within noise (not a regression, per the decision rule's
"flat-within-CI" allowance). **This 45.00%/41.75% pairing must NOT be read as "paired by
seed"** — per the methodology finding above, these are two independent 400-game samples.

**Kernel head-to-head leg (v1 vs v3, direct match) — UNAVAILABLE this cycle.** New
`scripts/build_kernel_head_to_head.py` (forked from the proven `build_kernel_vs_baseline.py`
self-contained-driver packaging, extended to install TWO search_scorer snapshots under aliased
module names `search_scorer_a`/`search_scorer_b`). **Three real kernel pushes all hung
identically** at `KernelWorkerStatus.RUNNING` with zero checkpoints and — critically — zero
fetchable log output at all (`kaggle kernels output` returned nothing, not even a partial log
line), across 1hr / 15min / 10min wait windows:

1. First push: hung 1hr, zero output.
2. Retry (fresh slug, same code): hung 15min, zero output — ruled out a one-off transient stall.
3. Retry with a targeted fix (`multiprocessing.get_context("spawn")` forced instead of the
   platform default, hypothesizing Kaggle's Linux container defaults to `fork`, and forking
   after the compiled `cg` engine bindings were initialized *twice* at module level — once per
   installed search_scorer snapshot, each independently calling `all_card_data()`/
   `all_attack()` — might leave forked workers deadlocked on their first engine call): hung
   10min, zero output. This rules out the fork-safety hypothesis specifically.

Local isolated testing of the exact same `driver.py` (the technique that found the original
`build_kernel_vs_baseline.py` packaging bugs) confirmed the packaging/game logic itself is
correct on this machine — it played 160/400 games with plausible ~50/50 results before being
intentionally killed. The complete absence of even one fetchable log line (not just zero
checkpoints) across all three attempts suggests the hang happens very early — plausibly before
the game loop starts at all — in a way not reproducible locally and not diagnosable further
through the `kaggle` CLI's `status`/`output` commands alone. **User-confirmed decision: proceed
to A4 without this leg**, treating it as a real, unresolved gap (this is exactly the
"highest-uncertainty new infra this cycle" risk flagged before building it) rather than
guessing further at kernel-compute expense. Flagged as follow-up work, not blocking this
decision.

**Spot-autopsy**: 20 losses sampled at random from v3's 233 local losses (`random.seed(0)`).
All structurally ordinary — reasonable decision counts (11-55), complete turn progressions
(turns 6-29), **zero time-guard activations** in the sample (consistent with the 0/9843 rate
above). No new failure mode found; every sampled loss matches already-known patterns (no
crashes, no degenerate/truncated games).

## Decision rule, applied

> Ship if flip rate collapsed AND v3 beats/matches v1 head-to-head AND v3-vs-baseline is up or
> flat-within-CI vs v1-vs-baseline AND zero time-budget violations AND no new failure modes.

- Flip rate (literal metric): could not collapse for any agent, by construction — superseded by
  the self-consistency metric, which shows a real, directionally-correct improvement (97.3% vs.
  95.8%, instability 0.0% vs. 1.7%). Genuine positive signal, but modest. ⚠️
- v3-vs-baseline vs. v1-vs-baseline (local, independent samples — see the seed-determinism
  finding above): CIs overlap substantially — flat-within-CI, not a statistically distinguishable
  regression — **but v3's own point estimate is numerically lower** (41.75% vs. 45.00%). ⚠️
- Mechanism: **mixed, not a collapse**. Attack-decline (9.2%→7.2%), evolve-decline
  (22.4%→19.1%), and tied-and-lost *rate* (40.8%→37.2%) all improved — but starvation rate
  (16.3%→19.2%) and tied-and-lost *absolute count* (73→80) both got worse. ❌ (does not meet
  "collapse" bar)
- Zero time-budget violations: 0/9843 time-guard activations across the full run. ✅
- Head-to-head vs. v1: **unavailable** — 3 real kernel pushes hung identically, root cause not
  resolved this cycle. Cannot be marked ✅. ❌ (unevaluable)
- New failure modes: none found in the 20-loss spot-autopsy. ✅

**Per the rule as given, and consistent with the same rigor v2's own gate applied under similar
circumstances: DO NOT SHIP this cycle.** The self-consistency improvement is real (this is a
genuine, structural fix, not a false lead), but two of the five criteria fail outright
(mechanism didn't collapse, head-to-head unevaluable) and a third is ambiguous (win rate flat
but numerically down) — nowhere close to the "all criteria pass" bar the rule requires. v1
remains the active Kaggle submission; no new submission was made this cycle.

## What's committed vs. what's shipped

**Committed to `main`**: everything — the v3 aggregation code in `agents/search_scorer.py`
(built on top of v2's already-committed-but-unshipped tie-break fix), `tests/test_v3_voting.py`,
`tools/compute_v3_sample_budget.py`, `tools/run_flip_rate_smoke_check.py`,
`scripts/build_kernel_head_to_head.py` (packaging logic proven correct locally, real-kernel
execution still unresolved), and this report. Worth preserving: the aggregation logic is
correct and unit-tested, the self-consistency improvement is real, and reverting real tested
work for no functional reason would just make a future cycle redo it.

**NOT shipped to Kaggle**: v1 remains the active submission. No `scripts/build_submission.py` /
`verify_submission.py` / `submit.py` step was run, `VERSIONS.md` has no new `## v3` entry, no
`mu_history.jsonl` dual-logging applies.

## Follow-up for a future cycle

1. **Resolve the head-to-head kernel hang.** Three consecutive identical hangs (zero fetchable
   log output, not just zero checkpoints) across three different mitigation attempts is a real,
   reproducible infrastructure gap, not bad luck. Next step likely needs either direct
   inspection of the Kaggle kernel's raw execution log via the web UI (not exposed through the
   `kaggle` CLI's `status`/`output` commands as currently used), or a structurally different
   packaging approach (e.g. two separate single-agent kernels instead of one dual-agent kernel,
   to isolate whether installing two `search_scorer` copies in one process is the actual cause).
2. **The mechanism result (starvation rate up, tied-and-lost count up) deserves its own
   investigation** before another voting-related cycle — N=8 may simply not be enough samples
   for decisions with genuinely high hidden-world-dependent uncertainty (not everything near-50/50
   collapses with a modest sample size via majority vote), or there may be a real remaining gap
   in how votes/score-sum interact with `evaluate()`'s existing near-tie behavior.
3. **The seed-non-determinism finding applies project-wide**, not just to this cycle — recorded
   as a new method lesson for Workstream C's CLAUDE.md compaction: "seeds passed to
   `tools/eval_arena.py --seed` / kernel drivers' `SEED_BASE` do not control game-engine
   determinism (`battle_start()` takes no seed argument) — treat separate 400-game runs as
   independent samples (compare via CI overlap), never as paired-by-seed."

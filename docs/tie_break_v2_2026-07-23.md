# v2: target-aware tie-break extension — implement, verify, DO NOT SHIP — 2026-07-23

Implements the top-ranked fix from `docs/near_tie_measurement_2026-07-23.md`: extend
`agents/search_scorer.py`'s tie-break to route ties by attacker-pipeline membership and
proximity-to-powered, not just option kind. **Verified at the 400-game gate. Per the decision
rule stated in the task, this cycle does NOT ship**: win rate moved favorably, but the
mechanism the fix specifically targets (attacker-starvation "tied-and-lost") did not collapse —
it moved slightly in the wrong direction. No agent change was submitted.

## 1. Reproduce: 3 failing unit tests

New `tests/test_tie_break.py`. Searched all 45 of v1's real ladder games (not just the 20-item
starvation-in-losses set behind the original "14/20" figure) for genuine same-kind attach ties
with different pipeline targets — found 169 candidates, hand-verified 3 with real, deterministic
(confirmed identical across resamples — attaching energy to our own Pokémon never depends on
the opponent's hidden hand) scores captured live via `choose_action`'s own `trace_fn`:

| Case | Real board state | Old picks | New picks (correct) |
|---|---|---|---|
| `87507277`@step 51 | Crustle already at 4 energy (deficit 0) ties with 3 Dwebbles (deficit 3 each) | target 70 (the already-full Crustle — wasted) | target 64 (a still-building Dwebble) |
| `87508450`@step 55 | Crustle A deficit 3 ties with Crustle B deficit 1 | target 67 (deficit 3 — worse) | target 69 (deficit 1 — closer to ready) |
| `87510208`@step 48 | same pattern, different game | target 67 (deficit 3) | target 69 (deficit 1) |

All 3 **confirmed FAILING** against pre-fix code (exact wrong targets: 70, 67, 67 — matching the
table above), then **confirmed PASSING** after the fix. `PYTHONIOENCODING=utf-8 py -3.14
tests/test_tie_break.py`.

## 2. Design (documented in `CLAUDE.md`, "SearchScorer's tie-break ordering")

Extends `_tie_break_key()`'s tuple from `(kind_priority, targets_unpowered_attacker)` to
`(kind_priority, pipeline_rank, proximity)`:
- `ATTACKER_PIPELINE_IDS` — computed once at import by walking the deck's designated attacker
  card's `evolvesFrom` chain (a **name** string per `cg.api.CardData`, matched back to an id) —
  not a second hardcoded tuple. Verified against `decks/crustle_wall_deck.csv`'s actual
  composition: exactly one Pokémon line (4x Dwebble `344`, 4x Crustle `345`, everything else
  Trainers/Energy), so this resolves to `{344, 345}` — the same ids `evaluate()`'s
  `turns_to_power` already uses, now derived rather than duplicated.
- Pre-evolutions count as the attacker they become (energy persists through evolution).
- Among pipeline targets, a member still needing energy is *preferred over* one already at
  cost (attaching there is a no-op — and `evaluate()`'s own `wasted_energy` feature is a
  `min`-across-pipeline feature that can't always catch this itself, since it stays 0 as long
  as *any* pipeline member is already full, regardless of which specific target this
  attachment goes to). Among still-needing members, smaller deficit (closer to ready) wins.
- `evaluate()` itself untouched — only which of several *already-equally-scored* options gets
  picked changes.

## 3. Implement + local verification

- Full existing test suite (`test_obs.py`, `test_baseline.py`, `test_search_scorer.py`,
  `test_encode.py`) plus the 3 new tests: **all pass**.
- **Important methodology finding from the 117-decision replay smoke check** (re-running
  `tools/measure_near_tie_hypothesis.py` against the fixed code): directly comparing old vs.
  new tie-break output on the *same real tied scores* across all 18 checkable decisions in the
  original 20-item starvation-in-losses set: **0/18 changed**. Investigated why: those 20
  historical decisions turn out to be dominated by single-clear-Crustle scenarios the *old*
  Crustle-only check already handled correctly — the 3 genuine failure cases above came from a
  broader search across all 45 games, in different games/turns than that specific 20-item set.
  **This was flagged to the user before the 400-game run** as a reason to temper expectations
  about how much the fix would move the historical "14/20" number specifically — confirmed
  prescient by the 400-game result below.

## 4. 400-game gate (real Kaggle kernel, both legs)

New `scripts/build_kernel_vs_baseline.py`. Building this surfaced and fixed **three real,
previously-undiscovered bugs** in this repo's Kaggle-kernel packaging approach (the *first* real
end-to-end use of it — `scripts/build_kernel_bakeoff.py` shares the same latent packaging bug
and has still never been exercised):
1. The competition-mounted SDK path has an extra `competitions/` segment
   (`/kaggle/input/competitions/pokemon-tcg-ai-battle/...`) the original guess list lacked.
2. **`kaggle kernels push` for a `kernel_type: "script"` does NOT upload sibling files** — a
   downloaded kernel log showed `/kaggle/src` containing only `['__script__.ipynb',
   'script.py']`. Every "copy these files into the kernel folder alongside driver.py" pattern
   (used by both this new script and the pre-existing `build_kernel_bakeoff.py`) silently drops
   everything except the designated `code_file`. Fixed by making the driver fully
   self-contained: every needed module's source is base64-embedded and installed into
   `sys.modules` via `exec()` at driver startup; `deck.csv` is embedded as a literal list.
3. `kaggle kernels output` captures `/kaggle/working`, not `/kaggle/src` (where the script
   itself runs) — output files are now written to an explicit `/kaggle/working` path.

Verified the fixed packaging in true isolation locally (only `driver.py` present, matching
Kaggle's real upload behavior exactly) before spending further real kernel time. Ran with
`--push --wait`; the checkpointing path (partial-result writes every 20 games) was exercised by
the failed earlier attempts but not needed on the successful run (both legs completed in one
push with 0 crashes).

## 5. Decision rule, applied

> Ship if tied-and-lost collapses AND winrate is up or flat-within-CI AND no new failure mode.
> If winrate drops beyond noise or the mechanism didn't move, stop and report — do not ship.

- Win rate: up, within noise of flat. ✅
- No new failure mode: none found in the spot-autopsy. ✅
- **Tied-and-lost collapses: NO — it went from 33.3% to 34.7% of starvation cases (66→75
  absolute), i.e. it did not move in the intended direction at all.** ❌

**Per the rule as given: DO NOT SHIP.** The fix is real, correctly implemented, and proven
correct in the 3 isolated reproduction cases — but at the scale of a real 400-game gate, it does
not measurably reduce the mechanism it targets. The likely explanation (see §3): the 3
reproduction cases represent a real but apparently minor slice of real starvation decisions;
most real starvation/tied-and-lost cases are evidently driven by something else the fix doesn't
touch — plausibly the cross-kind, determinization-sampling-driven near-ties
`docs/near_tie_measurement_2026-07-23.md`'s item 1 already identified as the dominant remaining
driver, which this cycle deliberately left untouched ("no sampling changes — that's a future
v3"). v1 remains the shipped submission; no new submission was made this cycle.

## What's committed vs. what's shipped (these are different questions)

**Committed to `main`**: everything, including the `agents/search_scorer.py` fix itself.
It's correct (3 isolated reproduction cases, all existing tests pass) and worth preserving in
history rather than discarding — reverting real, tested work for no functional reason would
just make a future v3 cycle redo it. `tests/test_tie_break.py`,
`scripts/build_kernel_vs_baseline.py` (now proven working end-to-end — the first Kaggle-kernel
verification tool in this repo to actually complete a real run), and `CLAUDE.md`'s tie-break
ordering doc (accurately describes the code that's now in `agents/search_scorer.py`) are
committed alongside it.

**NOT shipped to Kaggle**: **v1 remains the active submission.** No `scripts/build_submission.py`
/ `verify_submission.py` / `submit.py` step was run this cycle, `VERSIONS.md` has no new `## v2`
entry, and no `mu_history.jsonl` dual-logging applies (that was conditional on shipping). A
future cycle that wants to actually submit this fix (e.g., bundled with the sampling work in a
combined v3, or on its own after re-verifying) needs to explicitly re-run the ship pipeline —
the presence of the fix in `agents/search_scorer.py` is not itself a decision to submit it,
exactly as this report's mechanism data argues it shouldn't be, on its own, right now.

## Suggested next step (not this cycle)

Per `docs/near_tie_measurement_2026-07-23.md`'s own ranking, candidate (b) — more
determinization samples per decision (currently exactly 1; measured per-call cost ~10-50ms,
trivial against Kaggle's disabled per-move timeout) — was already flagged as the second
priority behind this fix. Given this fix's real-world effect on the mechanism turned out to be
small, a future cycle should measure candidate (b)'s effect on the SAME mechanism (starvation/
tied-and-lost rate, plus the broader attack-vs-setup near-tie flip rate from
`docs/near_tie_measurement_2026-07-23.md` item 1) before committing further engineering time to
target-selection refinements specifically.
